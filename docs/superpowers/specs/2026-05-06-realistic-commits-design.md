# 设计：提升自动提交脚本的可信度和真实感

## 目标

改造 `codeTask.sh`，使自动生成的提交在时间节奏、提交信息、代码变更方式三个维度更接近真实开发行为。

## 约束

- 只修改 `TS_FILES` 数组中的目标文件，绝不影响仓库中的其他文件
- 保持 Shell 脚本实现，不引入新依赖
- 与现有 LaunchAgent 配置兼容

## 设计

### 1. 提交时间随机化

脚本开头加随机延迟 0~900 秒。配合 LaunchAgent 的 40 分钟固定间隔，实际提交间隔变为 25~55 分钟。

```bash
RANDOM_DELAY=$((RANDOM % 901))
sleep $RANDOM_DELAY
```

### 2. 增量修改策略

每次执行随机选择一种操作类型，而非每次清空重写：

| 操作类型 | 基础权重 | 说明 |
|---------|---------|------|
| 追加新代码 | 40% | 在文件末尾添加接口/函数/类型 |
| 修改已有代码 | 25% | 改变量名、加参数、调整类型 |
| 删除旧代码 | 15% | 删除一个完整的 export 块 |
| 调整注释/格式 | 20% | 增删改注释、调整空行 |

文件增长控制：
- 超过 120 行：删除权重提升到 40%，追加降到 15%
- 低于 20 行：强制追加

实现方式：
- 追加：从预置代码片段随机选取追加到文件末尾
- 修改：`sed` 对已有变量名/类型做随机替换
- 删除：`sed` 定位完整 `export` 块整块删除
- 注释：`sed` 在随机行插入/替换/删除注释

### 3. 提交信息生成

根据实际执行的修改类型动态生成，保持全中文：

| 操作类型 | 对应前缀 |
|---------|---------|
| 追加新代码 | `feat:` |
| 修改已有代码 | `fix:` 或 `refactor:` |
| 删除旧代码 | `refactor:` |
| 调整注释/格式 | `style:` 或 `chore:` |

每个文件 × 每种操作准备 4~5 条模板，总共约 80~100 条。提交信息与实际变更类型一致。

示例：
- tracking.ts + 追加 → "feat: 添加页面停留时长追踪接口"
- validation.ts + 修改 → "fix: 修复手机号校验正则匹配问题"
- logger.ts + 注释 → "style: 统一日志模块注释格式"

## 文件变更范围

仅修改 `codeTask.sh` 一个文件。不新增文件，不修改 LaunchAgent 配置。
